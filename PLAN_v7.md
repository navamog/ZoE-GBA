# ZOE GBA 한글화 계획 (v7) — 2바이트 폰트 확장

## 배경 / 강좌 참고
- guide/동키콩2 강좌: "폰트코드 1→2바이트 확장" 기법, 상대주소 포인터(기준점+값×4), LZ77 대사 ROM 이식
- guide/타일맵 강좌: VRAM 여유공간 타일 추가 + DMA3 전송크기/소스 패치 + 타일맵 수정
- ZOE는 텍스트가 **ROM에 비압축 평문**으로 저장 → 동키콩2의 LZ77 문제 없음(in-place 덮어쓰기 가능)

## 핵심 진단 (정량)
- 번역에 필요한 **고유 한글 음절 = 845개**
- v6까지의 방식: 단일바이트 0x80~0xFE = **127슬롯이 물리적 상한** → 고유음절 15%, 출현 23% 누락 → 깨짐

## ROM 분석 결과 (capstone 정밀 분석)
- 텍스트 파서 루프 0x03DC8~: `ldrb [r5]`로 1바이트 읽기
  - `0x00`=종료, `0x01~0x0F`=**2바이트 리드바이트**(code=byte0<<8|byte1, 0x100+ → 그리기 경로 직행), `0x10~0x1F`=제어 점프테이블, `0x20~0xFF`=단일문자
  - → **파서가 이미 2바이트 문자(0x0100~0x0FFF, 최대 3840코드)를 native 지원**. 파서 수정 불필요.
- 글리프 함수 0x38AC (호출처 단 1곳 0x3558): `return glyph_base + charcode*32`. 반환 포인터를 호출자 0x35BE가 `ldrb [r7]`로 직접 읽어 픽셀 플롯 → **ROM 글리프 포인터 직접 반환이 유효**
- 글리프 포맷: 32바이트 = 16행×2바이트(빅엔디안, **MSB=좌측 픽셀**)
- 너비 함수 0x3900: table1=0x553676(cache≥0), table2=0x553476(cache<0), 마스크 0x1FF.
  **코드>0x1FE & cache≥0 → 자동 12px** (테이블 패치 불필요)
- 호출자가 같은 slot으로 글리프·너비 함수를 연속 호출 → 글리프 훅에서 `IWRAM[slot*32+31]=0` 세팅 시 너비도 12px 확정
- US(영문) ROM 표시 텍스트는 순수 ASCII(byte=ord-6, space=0x20) → 0x01~0x0F 리드바이트 미사용 → 2바이트 영역 한글 점유 안전

## v7 설계
- 한글 음절 i(0..844) → **문자코드 0x200+i** (리드 0x02~0x05, 모두 >0x1FE)
- 텍스트: 각 음절을 빅엔디안 2바이트 `[code>>8, code&0xFF]`로 인코딩
- 훅 0x038AC(8B) → 트램폴린(0x78C540): 코드 0x200~0x5FF면 `IWRAM[slot*32+31]=0` 후 `GLYPH_GBA+(code-0x200)*32` 반환, 아니면 원본 경로(0x38AC 4명령 재현 후 0x38B4 복귀)
- 글리프: Galmuri11.bdf(11px 비트맵, 845음절 100% 커버) → 16×16 1bpp, ROM 0x78C580~
- 너비 패치 불필요(자동 12px)

## 한계 / 향후 (강좌 기법 적용 단계)
- in-place 덮어쓰기라 한글 2바이트가 원문 영문 바이트길이를 초과하면 truncate됨
- 완전 해결: 동키콩2 강좌처럼 텍스트를 확장 ROM 영역으로 이식 + 포인터 테이블 재지정 (다음 단계)

## 검증
1. 트램폴린/훅을 capstone으로 역어셈블해 오프셋·풀주소 정합 확인
2. mGBA로 ZOE_Korean_v7.gba 실행 → 인트로/대사에서 한글 표시 확인
